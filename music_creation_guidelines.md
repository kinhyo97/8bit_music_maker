# 음악 제작 지침

이 문서는 8bit music maker에서 새 곡을 만들거나 기존 곡을 수정할 때 따르는 기준이다. 목표는 단순히 음을 찍는 것이 아니라, 게임에서 바로 쓸 수 있는 루프 BGM을 빠르게 만들고 피드백에 맞춰 계속 다듬을 수 있는 구조를 유지하는 것이다.

## 1. 기본 방향

곡은 항상 `LoopSpec` 구조를 기준으로 만든다.

- `lead`: 메인 멜로디 또는 가장 잘 들리는 상단 라인
- `bass`: 저역 진행과 루프의 무게 중심
- `arp`: 보조 리듬, 반짝이는 장식, 고역 강조
- `chords`: 마디별 화성 흐름
- `drums`: 킥, 스네어, 하이햇 리듬

새 곡을 만들 때는 사운드부터 고르기보다 먼저 다음을 정한다.

1. 곡의 용도: 메뉴, 필드, 전투, 보스전, 긴장, 감성, 재즈, 클래식 등
2. 마디 수: 짧은 루프는 4마디, 전개감이 필요하면 8마디
3. BPM: 느린 분위기는 80~110, 경쾌한 게임곡은 120~150, 빠른 곡은 160 이상
4. 키와 스케일: 기본은 C major/A minor, 필요하면 다른 키 사용
5. 코드 진행: 멜로디보다 먼저 분위기를 잡는 뼈대

## 2. 루프 기준

게임에서 쓸 곡은 반복될 때 어색하지 않아야 한다.

- 마지막 마디가 첫 마디로 자연스럽게 돌아오게 만든다.
- 마지막 음을 너무 길게 끌어서 루프 시작과 겹치게 만들지 않는다.
- 강한 마무리를 넣더라도 첫 박으로 이어지는 긴장감을 남긴다.
- 4마디 루프는 단순하고 기억하기 쉽게 만든다.
- 8마디 루프는 1~4마디와 5~8마디에 약간의 변화를 둔다.

루프가 어색하면 멜로디보다 먼저 베이스와 코드 진행을 확인한다. 대부분의 루프 어색함은 마지막 코드가 첫 코드로 돌아가는 방식에서 생긴다.

## 3. 시간 입력 기준

모든 음은 `bar`와 `beat` 기준으로 찍는다.

- 한 마디는 4박이다.
- 16분음표는 `0.25` 단위로 찍는다.
- 8분음표는 `0.5` 단위로 찍는다.
- 마디 끝 네 칸 기준 첫 칸과 세 번째 칸은 `beat: 4`, `beat: 4.5`다.

예시:

```ts
{ bar: 2, beat: 4, note: "C#5", length: "16n" }
{ bar: 2, beat: 4.5, note: "G#5", length: "16n" }
```

빠른 악보 기반 패턴은 직접 이벤트를 하나씩 쓰지 말고 헬퍼를 사용한다.

```ts
...notesToEvents(1, ["G#4", "C#5", "E5", "G#5"])
```

## 4. 멜로디 작성 기준

멜로디는 무작위로 찍지 않는다. 규칙 기반으로 만들되, 너무 같은 패턴만 반복하지 않도록 한다.

좋은 멜로디는 보통 다음 중 하나를 가진다.

- 짧은 모티프를 반복하면서 조금씩 변형
- 코드톤 중심 진행
- 마디 끝에서 다음 마디로 끌고 가는 음
- 한 번 올라갔으면 내려오는 호흡
- 같은 리듬 안에서 음만 바꾸는 반복감

8비트 게임 음악에서는 멜로디가 너무 복잡하면 피곤해진다. 빠른 곡일수록 음색은 단순하게 두고, 리듬과 코드 진행으로 긴장감을 만든다.

## 5. 악보 참고 기준

악보를 참고할 때는 먼저 곡의 핵심 구조를 파악한다.

- 오른손이 멜로디인지, 분산화음인지 확인한다.
- 왼손이 베이스 반복인지, 코드 진행인지 확인한다.
- 강하게 찍는 악센트 위치를 먼저 찾는다.
- 원곡의 모든 음을 복사하려고 하기보다, 루프 엔진에 맞는 핵심 패턴부터 옮긴다.

클래식처럼 퍼블릭 도메인인 곡은 비교적 가깝게 참고할 수 있다. 다만 앱 구조상 피아노 양손 전체를 그대로 옮기기보다 `lead`, `bass`, `arp`, `chords`에 역할을 나눠 배치한다.

현대 곡이나 저작권이 있는 곡은 그대로 복제하지 않는다. 리듬감, 진행 방향, 분위기만 참고하고 새 멜로디로 만든다.

## 6. 코드 진행 기준

코드는 곡의 분위기를 가장 빨리 바꾼다. 멜로디가 애매하면 먼저 코드 진행을 수정한다.

기본 진행:

```ts
["C", "G", "Am", "F"]
["Am", "F", "C", "G"]
["C#m", "G#7", "C#m", "G#7 C#m"]
```

재즈나 세련된 느낌:

```ts
["Cmaj9", "A7b9", "Dm9 G13", "Cmaj7#11"]
```

한 마디에 코드가 두 개 필요하면 문자열 안에 공백으로 나눠 넣는다.

```ts
"Dm9 G13"
```

텐션 코드는 과하게 넣지 않는다. 멜로디가 복잡할수록 코드는 단순하게, 멜로디가 단순할수록 코드는 조금 더 화려하게 잡는다.

## 7. 드럼 작성 기준

드럼은 처음부터 과하게 넣지 않는다.

기본 순서:

1. 킥으로 박자 중심을 잡는다.
2. 스네어로 강세 위치를 만든다.
3. 하이햇으로 속도감과 빈 공간을 채운다.
4. 필요할 때만 추가 킥이나 스네어를 넣는다.

게임 루프에서는 드럼이 너무 복잡하면 멜로디를 방해한다. 특히 빠른 16분음표 멜로디가 있을 때는 하이햇을 살짝 줄이고 킥/스네어의 위치만 명확하게 둔다.

강하게 내리찍는 부분은 킥과 스네어를 동시에 넣고, 가능하면 lead나 arp에도 코드톤을 겹친다.

```ts
drums: {
  kick: [{ bar: 1, beat: 4 }, { bar: 1, beat: 4.5 }],
  snare: [{ bar: 1, beat: 4 }, { bar: 1, beat: 4.5 }],
  hat: [...]
}
```

## 8. 악기 선택 기준

현재 사용할 수 있는 악기 타입:

- lead: `square`, `sawtooth`, `triangle`, `pluck`
- bass: `triangle`, `square`
- arp: `square`, `pulse`, `pluck`
- drums: `noise`

권장 사용:

- `square`: 8비트 메인 리드에 가장 안정적
- `triangle`: 부드러운 베이스나 낮은 음역에 적합
- `sawtooth`: 거칠고 강한 리드에 적합
- `pluck`: 짧고 뜯기는 소리, 아르페지오나 장식에 적합
- `pulse`: 칩튠 아르페지오 느낌에 적합

빠른 클래식 분산화음은 `square`가 선명하다. `pluck`은 재미있지만 너무 튀면 메인 리드보다 보조 아르페지오에 쓰는 편이 낫다.

## 9. 곡 수정 피드백 반영 기준

사용자 피드백은 음악적 판단으로 번역해서 반영한다.

- "킥이 약하다": 볼륨보다 저역 바디와 펀치 레이어를 먼저 확인
- "하이햇이 작다": 볼륨, 필터, 패턴 밀도를 확인
- "너무 정신없다": arp나 hat을 줄이고 lead를 살림
- "더 신나게": BPM, 킥 패턴, 밝은 코드톤, 상행 멜로디 확인
- "더 악보처럼": lead의 음 진행과 beat 위치를 먼저 수정
- "두 번 찍어라": 정확한 beat 위치를 확인하고 chordHits/drums를 같이 반영

피드백이 들어오면 전체를 갈아엎기보다 가장 영향이 큰 레인부터 수정한다.

1. 리듬 문제: drums
2. 음 진행 문제: lead
3. 분위기 문제: chords
4. 무게감 문제: bass
5. 반짝임 문제: arp

## 10. 샘플 확장 기준

나중에 샘플을 추가할 때도 곡 데이터 구조는 유지한다. 바뀌는 것은 소리를 내는 엔진뿐이다.

드럼 샘플 권장 구조:

```text
samples/drums/basic/kick.wav
samples/drums/basic/snare.wav
samples/drums/basic/hat_closed.wav
samples/drums/basic/hat_open.wav
```

악기 샘플 권장 구조:

```text
samples/instruments/pluck/C3.wav
samples/instruments/pluck/E3.wav
samples/instruments/pluck/G3.wav
samples/instruments/pluck/C4.wav
```

샘플 파일 기준:

- 앞 무음 제거
- 피크가 터지지 않게 정규화
- `wav`, `44.1kHz`, `16bit` 또는 `24bit` 권장
- 파일명에는 `#` 대신 `s` 사용 권장: `D#4`보다 `Ds4`
- tail이 너무 긴 샘플은 루프에서 지저분해질 수 있음

## 11. 검증 기준

곡을 수정한 뒤에는 항상 다음을 확인한다.

- `npm run build` 통과
- 곡 리스트에서 정상 로드
- Play/Stop 동작
- Space 키 토글 동작
- Loop 켰을 때 반복이 자연스러운지
- WAV 다운로드가 깨지지 않는지

오디오 테스트를 위해 브라우저를 띄웠다면 반드시 마지막에 재생을 멈추고 브라우저 세션을 닫는다.

## 12. 다음 확장 방향

우선순위는 다음과 같다.

1. 드럼 샘플팩 구조 추가
2. 사운드팩 선택 UI 추가
3. lead/arp/bass 샘플 악기 추가
4. 곡마다 악기 프리셋 저장
5. intro + loop body 구조 분리
6. 악보 기반 입력 헬퍼 강화
7. 코드 진행 추천 또는 자동 보이싱 추가

지금 엔진의 핵심은 곡 데이터와 사운드 출력이 분리되어 있다는 점이다. 따라서 앞으로 기능을 추가할 때도 `LoopSpec`을 쉽게 버리지 않고, 엔진과 UI를 교체 가능한 방식으로 확장한다.
